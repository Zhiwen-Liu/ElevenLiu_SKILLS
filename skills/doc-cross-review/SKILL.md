# 迁移文档交叉审查

## 概述

本技能对硬件迁移项目（如 CPU → NPU/GPU）的对比验证文档进行系统性交叉审查。
审查目标不是重新运行测试，而是通过文档-代码-文档之间的三角交叉验证，发现事实性错误、模块间矛盾和逻辑缺陷。

## 核心原则

### 原则 1: 文档-代码双向校验

文档中的每一个定量声明都必须能在代码中找到对应来源，反之亦然。

**操作步骤：**

```bash
# 1. 扫描代码中所有数值断言，建立真实覆盖清单
grep -rn 'allclose\|isclose\|atol\|assert.*abs' tests/ scripts/

# 2. 与文档覆盖映射表逐行比对
#    - 文档声称的测试文件是否真的是数据来源？
#    - 数据来自 tests/ 还是独立验证脚本？
#    - 两者的测试条件（seed、mask、维度）是否一致？
```

典型矛盾模式：
- 文档覆盖表将 `test_xxx.py` 映射到某章节，但该章节数据实际来自 `verify_precision.py`
- 测试代码用 `randint(0,1)` 产生全零 mask（trivial case），而验证脚本用 `randint(0,2)` 产生非零 mask（真实 case），文档未区分

### 原则 2: 总结-细节一致性校验

文档的「总结表」与「细节章节」之间的数据必须严格一致。

**检查项：**

| 检查维度 | 典型矛盾 |
|---------|----------|
| 数量 | 总结说 "4 项指标一致"，细节只列了 2 项 |
| 覆盖 | 总结列 5 个组件，细节表有 6 个（遗漏 1 个） |
| 数值 | 总结表四舍五入后与细节表不一致 |
| 结论 | 总结标 PASS，细节数据实际显示 DIFF |

操作方法：逐行提取总结表的每个声明，在细节章节中找到对应数据，核实数值和结论匹配。

### 原则 3: 多文档间交叉校验

同一项目的多份文档（如 `CPU_vs_NPU.md` 和 `MIGRATION_REPORT.md`）之间数据必须一致。

```bash
# 提取两份文档中的关键数字并 diff
grep -n '加速比\|speedup\|ms\|PASS\|FAIL' doc1.md > /tmp/d1.txt
grep -n '加速比\|speedup\|ms\|PASS\|FAIL' doc2.md > /tmp/d2.txt
# 人工比对
```

常见问题：
- 性能数据在两处各出现一次，更新了一处忘了另一处
- 适配文件数、测试通过数等统计口径不一致

### 原则 4: 逻辑自洽性审查

审查文档中的因果推理链是否自洽，特别关注：

**4a. RNG 差异叙事一致性：**
如果文档解释 "模型 A 的差异来自 RNG 后端不同"，那么使用相同 RNG 机制的模型 B 也应有差异。
如果 B 差异为零，必须有额外解释，否则构成自相矛盾。

验证方法：
```bash
# 检查两个模型是否共享同一采样路径
grep -n 'torch.randn\|random_rotations\|random_quaternions' src/model/modules/diffusion.py
# 如果共享，则 RNG 差异应同时影响两者
```

**4b. 交叉验证链的逻辑严密性：**
验证链中每一层证明的是什么维度：
- 同设备 chunk 一致性 → 证明 NPU 数值稳定性，**不是** CPU vs NPU 精度
- 跨设备算子对比 → 证明基础精度对齐
- 跨设备端到端对比 → 证明整体精度对齐

不同维度的验证不能混入同一条链。

**4c. 性能拆分自洽性：**
如果文档声称 "固定开销与设备无关"，用端到端数据减去纯计算数据，CPU 和 NPU 的固定开销应近似相等。
```
CPU固定开销 = CPU端到端 - CPU纯计算
NPU固定开销 = NPU端到端 - NPU纯计算
若两者差距 > 20%，说明前提不成立或拆分有误。
```

### 原则 5: 复现命令完整性校验

复现命令必须能完整重现文档中声称的测试条件。

```bash
# 提取文档声称的测试条件
# 例如: "recycling=1, sampling=5, seed=42"

# 提取复现命令中的实际参数
# 例如: "boltz predict input.yaml --no_kernels --seed 42"

# 检查代码中的默认值
grep -n 'recycling_steps\|sampling_steps' src/main.py
# 如果默认值与文档声称条件不同，且复现命令未显式指定，则为错误
```

### 原则 6: 代码统计口径校验

文档中的文件数、测试数等统计声明，用代码实际计数验证。

```bash
# 验证 "适配 N 个文件"
grep -rl 'from xxx.npu_utils' src/ --include='*.py' | wc -l

# 验证 "M passed, K sub-tests"
# 数测试方法和 subTest 数量
grep -rn 'def test_\|subTest' tests/ --include='*.py'

# 验证 "N 阶段 CPU vs NPU"
# 数 report() 调用次数
grep -c 'report(' scripts/verify_precision.py
```

### 原则 7: 精度数据 vs 阈值一致性校验

文档声称的 PASS 必须能被实际数据和实际阈值支撑。不能出现 "数据超过阈值但标 PASS" 的情况。

**检查项:**

| 检查 | 典型问题 |
|------|----------|
| max_abs vs atol | 文档说 atol=1e-4，但 max_abs=1.59e-3，却标 PASS |
| 判定公式 | 文档简化为 "max_abs < atol"，但脚本实际用 np.allclose(atol, rtol)，靠 rtol 补偿通过 |
| atol 合理性 | atol 设得太小导致实际靠 rtol 补偿，判定标准名不副实 |

操作步骤：
1. 提取脚本中每个 report() 的 atol 值
2. 对每个算子，检查: max_abs_diff <= atol ? 若否，检查是否靠 rtol 补偿
3. 文档中的判定标准描述必须与脚本实际用的公式一致

### 原则 8: 文档格式规范性检查

检查文档中是否存在可能导致渲染问题或可读性问题的特殊字符和格式问题：

- 特殊符号：不使用非常规 Unicode 符号（如 section sign 等）替代中文
- 代码块闭合：所有 ``` 必须成对出现
- 表格对齐：Markdown 表格的列数必须一致


## 工作流程

```
1. 通读所有文档，提取全部定量声明
2. 扫描代码，建立真实覆盖清单（断言数、seed、参数）
3. 总结表 vs 细节表逐行比对
4. 多文档间数据交叉比对
5. 逻辑链自洽性审查（RNG、验证链、性能拆分）
6. 复现命令 vs 测试条件 vs 代码默认值三方比对
7. 统计数字用代码实际计数验证
8. 按严重程度分级汇总（🔴事实错误 / 🔴逻辑矛盾 / 🟡数据不一致 / ℹ️表述可改进）
```

## 输出格式

审查意见保存为 `docs/<原文档名>_review.md`，按以下结构组织：

```markdown
# <原文档名> 审查意见

审查对象：<文件列表>

## 🔴 事实性错误
### 1. <标题>
- **位置**：<文件:行号或章节>
- **问题**：<具体描述>
- **修复建议**：<如何修复>

## 🔴 逻辑矛盾
### N. <标题>
...

## 🟡 数据不一致
### N. <标题>
...

## ℹ️ 表述可改进
### N. <标题>
...
```

每条问题必须包含：位置（可定位到文件和章节）、问题描述（含具体数据）、修复建议。
